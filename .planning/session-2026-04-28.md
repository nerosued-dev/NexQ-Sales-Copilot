# Session Handoff — 2026-04-28

## Objetivo
Agregar modo de transparencia real al overlay de NexQ para ver otras apps detrás (como Cluely/OpenCluely).

## Estado actual: EN PROGRESO

### Lo que se hizo (en orden)

#### 1. ConfigStore — `overlayOpacity`
- Agregado `overlayOpacity: number` (default `0.65`) al store
- Setter `setOverlayOpacity`, persistido en `config.json`
- `onKeyChange("overlayOpacity")` para sync cross-window
- Cargado en `loadConfig()` con fallback `0.65`

#### 2. UI — Botón ojo + slider
- **Botón ojo** en header del `OverlayView`: cicla `[0.9, 0.65, 0.35, 0.1]`
- **Slider** en Settings → General "Overlay Transparency" (10%–100%)
- Tooltip muestra porcentaje actual

#### 3. Diagnóstico de arquitectura (KEY FINDING)
El meeting UI NO corría en el overlay Tauri window (`transparent:true`).  
Corría en el **launcher window** (no transparente).  
El overlay Tauri window estaba siempre oculto para meetings desde UI.

#### 4. Fix arquitectural — window routing
**`src/App.tsx`:**
- Detecta window label (`windowLabel = "launcher"` | `"overlay"`)
- **Launcher**: cuando `currentView === "overlay"` → muestra overlay Tauri window + se oculta
- **Launcher**: escucha `nexq:meeting_ended` → resetea estado + se muestra
- **Overlay**: escucha `nexq:meeting_started` → inicializa estado local + render OverlayView
- **Overlay**: cuando `currentView → "launcher"` (endMeetingFlow) → se oculta + muestra launcher
- Overlay window ignora `firstRunCompleted` y `configLoaded` para renderizar

**`src/stores/meetingStore.ts`:**
- `startMeetingFlow`: emite `nexq:meeting_started` con `{meeting, audioMode, aiScenario}`
- `endMeetingFlow`: emite `nexq:meeting_ended`

**`src/components/ActiveMeetingProvider.tsx`:**
- Prop `isLauncherWindow: boolean`
- Hooks de persistencia/audio solo en launcher → evita duplicate DB writes

#### 5. Transparencia WebView2
**`src/App.tsx`:**
- CSS body/html → `transparent` para overlay window
- `setBackgroundColor([0,0,0,0])` en WebView2
- `Effect.Blur` (JS) → `DwmEnableBlurBehindWindow`

**`src-tauri/src/lib.rs`:**
- `apply_blur(&overlay, Some((0,0,0,0)))` desde Rust en `show_overlay()`

**`src-tauri/Cargo.toml`:**
- `window-vibrancy = "0.5"` agregado como dependencia

**`src-tauri/capabilities/default.json`:**
- Permisos agregados: `allow-set-background-color`, `allow-set-webview-background-color`, `allow-set-decorations`, `allow-set-skip-taskbar`, `allow-set-effects`, `allow-set-always-on-top`

#### 6. Settings fix en overlay
Cuando el usuario hace click en Settings durante meeting → `currentView = "settings"` → overlay renderizaba blank.  
Fix: overlay window también renderiza `<SettingsOverlay>` cuando `currentView === "settings"` o `settingsOpen`.

#### 7. Rust warning fix
`embedder.rs`: eliminado `last_err` variable innecesaria, `return Err(...)` directo en el loop.

## Problemas pendientes / por confirmar

### Transparencia — NO confirmada que funcione
- El usuario reporta que al bajar opacity solo ve blanco/gris (no otras apps)
- Posibles causas:
  1. `Effect.Blur` en Windows 11 puede no mostrar otras apps (solo wallpaper)
  2. `apply_blur` desde Rust funciona solo cuando se llama desde tray, no desde UI
  3. Race condition: `nexq:meeting_started` puede llegar antes que el listener esté listo
  4. El `nexq:meeting_started` evento puede no estar llegando al overlay window

### Próximos pasos si transparencia sigue sin funcionar
1. **Verificar que el overlay Tauri window está siendo usado**: agregar `console.log("[overlay] windowLabel:", label)` al inicio del useEffect en App.tsx para confirmar
2. **Verificar que el evento llega**: agregar `console.log("[overlay] nexq:meeting_started received")` en el listener
3. **Alternativa**: usar `window-vibrancy::apply_blur` también cuando el overlay window es mostrado desde JS (no solo desde tray/Rust) — añadir un Tauri command `apply_overlay_blur` que el frontend llame
4. **Alternativa nuclear**: en lugar de `Effect.Blur` probar con CSS `opacity` al nivel del window via `getCurrentWebviewWindow().setOpacity()` (si existe en la versión usada)
